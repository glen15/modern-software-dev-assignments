# 코드 리뷰 체크리스트

## 개요

효과적인 코드 리뷰를 위한 체크리스트입니다.
이 체크리스트를 참고하여 일관성 있는 리뷰를 수행하세요.

---

## 1. 기능성 (Functionality)

### 요구사항 충족

- [ ] 요구사항/티켓에 명시된 기능을 구현했는가?
- [ ] 엣지 케이스를 고려했는가?
- [ ] 기존 기능에 영향을 주지 않는가?

### 로직 검증

- [ ] 비즈니스 로직이 올바른가?
- [ ] 조건문/반복문이 의도대로 동작하는가?
- [ ] 반환값이 예상과 일치하는가?

---

## 2. 보안 (Security)

### 입력 검증

- [ ] 사용자 입력을 검증하는가?
- [ ] SQL 인젝션 방지 (파라미터화된 쿼리)?
- [ ] XSS 방지 (출력 이스케이프)?
- [ ] 경로 탐색 방지?

### 인증/인가

- [ ] 적절한 인증이 적용되었는가?
- [ ] 권한 검사가 올바른가?
- [ ] 민감한 데이터가 노출되지 않는가?

### 비밀 정보

- [ ] 하드코딩된 비밀번호/API 키가 없는가?
- [ ] 환경 변수를 올바르게 사용하는가?
- [ ] 로그에 민감 정보가 출력되지 않는가?

---

## 3. 가독성 (Readability)

### 명명 규칙

- [ ] 변수/함수/클래스 이름이 명확한가?
- [ ] 일관된 명명 규칙을 따르는가?
- [ ] 약어 사용이 적절한가?

### 코드 구조

- [ ] 함수가 한 가지 일만 하는가?
- [ ] 함수 길이가 적절한가? (20줄 이하 권장)
- [ ] 중첩이 깊지 않은가? (3단계 이하 권장)

### 주석

- [ ] 필요한 곳에 주석이 있는가?
- [ ] 불필요한 주석이 없는가?
- [ ] 주석이 코드와 일치하는가?

---

## 4. 유지보수성 (Maintainability)

### 중복 제거

- [ ] 중복 코드가 없는가?
- [ ] 공통 로직이 함수로 추출되었는가?

### 의존성

- [ ] 불필요한 의존성이 없는가?
- [ ] 순환 의존성이 없는가?

### 확장성

- [ ] 변경하기 쉬운 구조인가?
- [ ] 매직 넘버 대신 상수를 사용하는가?

---

## 5. 성능 (Performance)

### 효율성

- [ ] 불필요한 연산이 없는가?
- [ ] 적절한 자료 구조를 사용하는가?
- [ ] N+1 쿼리 문제가 없는가?

### 리소스 관리

- [ ] 파일/연결이 제대로 닫히는가?
- [ ] 메모리 누수 가능성이 없는가?

---

## 6. 에러 처리 (Error Handling)

### 예외 처리

- [ ] 예외가 적절히 처리되는가?
- [ ] 너무 넓은 예외 처리가 없는가? (`except:`)
- [ ] 에러 메시지가 명확한가?

### 실패 시 동작

- [ ] 실패 시 시스템이 안전한 상태를 유지하는가?
- [ ] 트랜잭션이 적절히 롤백되는가?

---

## 7. 테스트 (Testing)

### 테스트 존재

- [ ] 새 코드에 대한 테스트가 있는가?
- [ ] 테스트가 통과하는가?

### 테스트 품질

- [ ] 테스트가 의미 있는 케이스를 다루는가?
- [ ] 엣지 케이스 테스트가 있는가?
- [ ] 테스트가 독립적인가?

## 리뷰 작성 가이드

### 좋은 피드백 예시

```
✅ 좋은 피드백:
"이 함수는 사용자 검증과 데이터 저장 두 가지 일을 합니다.
 validateUser()와 saveUser()로 분리하면 테스트와 유지보수가 쉬워질 것 같습니다."

❌ 나쁜 피드백:
"함수가 너무 깁니다."
```

### 피드백 유형

| 접두사 | 의미 | 예시 |
|--------|------|------|
| `[필수]` | 반드시 수정 필요 | 보안 취약점, 버그 |
| `[권장]` | 수정 권장 | 가독성, 성능 개선 |
| `[질문]` | 이해 확인 | 의도 파악 |
| `[칭찬]` | 좋은 코드 인정 | 모범 사례 |
| `[참고]` | 정보 공유 | 관련 문서, 패턴 |

### 피드백 작성 원칙

1. **구체적으로**: 어디가 문제인지 명확히
2. **이유와 함께**: 왜 변경이 필요한지 설명
3. **대안 제시**: 가능하면 해결책 제안
4. **존중하는 태도**: 코드를 비판, 사람을 비판 X

## 리뷰 예시

### 보안 관련

```python
# 리뷰 대상 코드
query = f"SELECT * FROM users WHERE name = '{name}'"

# 피드백
[필수] SQL 인젝션 취약점이 있습니다.
f-string 대신 파라미터화된 쿼리를 사용해주세요:

query = "SELECT * FROM users WHERE name = :name"
result = db.execute(query, {"name": name})
```

### 가독성 관련

```python
# 리뷰 대상 코드
def process(d):
    if d:
        if d.get('status') == 'active':
            if d.get('type') == 'A':
                return d['value'] * 2
    return 0

# 피드백
[권장] Guard clause를 사용하면 가독성이 좋아질 것 같습니다:

def process(data):
    if not data:
        return 0
    if data.get('status') != 'active':
        return 0
    if data.get('type') != 'A':
        return 0
    return data['value'] * 2
```

### 테스트 관련

```python
# 피드백
[질문] 이 함수에 대한 테스트가 없는 것 같은데,
테스트 추가 계획이 있으신가요?
특히 빈 리스트 입력 케이스가 중요해 보입니다.
```
